Monitoring API

Below documentation describes SEMSTORM API v3 part used for Monitoring functionallity. It is divided into sections describing each part of Monitoring.

We encourage to use SEMSTORM PHP SDK library which delivers simple and ready-to-use functionallity for SEMSTORM API.
Take into consideration that SEMSTORM PHP SDK library is under Apache License 2.0, it is distributed without any warranties, and support is not included. If you encounter any problems please leave a ticket to let us know.

Introduction

Get access token

Access token is unique set of characters which identifies you, and allow you to use API.

If you are registered user, you can go to "Services token" tab in SEMSTORM system while logged in, and simply generate new token for yourself.

Access token allow to benefit from all API functionallity in the name of user who generated it, so dont share it with any third party! It should be handled with caution like passwords.

If you feel unsafe with your token - eg. you suspect that token could be obtained by unauthorised persons, you can regenerate your token, and get new one in same page you get your first. Just remember that previous token will be expired in moment you generate new one, so you have to replace it in all your scipts and functions which was using old one.

API results structure

Results have uniform structure. After calling API (in any way) client will receive response, if call was made using SEMSTORM PHP SDK API it will be php array, if calling directly by curl with .json prefix it will be encoded JSON string, eventhou every response will have structure which allows it's easy itnerpretation and data retrieve. Response array may contain:

"params": variable (array/assoc array) containing request aprameters - it will be params set by client and send to server, or -if optional- added by server automatically.

"result": variable (array/assoc array) containing request result(s) if no error occured.

"error": variable (array/assoc array) containing request info about any errors which occured, it dont exist if request was succesful. It have "message" field which holds information about reason of failure.

"_credits": variable (int) appears for most of results, showing how much request costs, it can be zero.

"_credits_left": variable (int) appears for most of results, showing how many credits left for current account.

API restrictions

API have multiple limitations to maximize efficiency and data credibility.

Requests per second. Each account (access token) have limit in number of requests. This is represented by number of request per short time frame. It return error with code 503 and mesasge "You are requesting too fast." (see below).

API throttle prevention. Retrieving big sets of data, and/or setting date range long into past highly increase chance to get "Prevent throttling" error. In that case you can try again after some time, or try to divide request to few less and retrieve it piece by piece. It return error with code 503 and mesasge "Throttle prevention." (see below).

Single request limits. Those limits are set from above in some requests to prevent large calculations and data transfering. Mostly it applies to request like creation multiple objects, retireve sets of data. To prevent those errors reduce amount of data in request. It return error with code 400 and mesasge "Request limit.".

Error message

If error occurs response will be send with proper http code (all success responses have code 200), and carry error information in body. Encoded body will contain variable keyed as "error" in which will be placed (at least this one) "message" string. This string have error code, and information text.

If you encounter any error which is not listed as standard API error let us know.

See below possible error codes:

(400) Wrong input. Invalid or missing input data. Mostly additional info about requirements is displayed alongside error message.

(403) Access denied. Action you want to perform is impossible due to protection level.

(404) No entry for given id. Id dont exist - or is unavailable for current user.

(404) Undefined action. Callback was called but cannot determine which action to take.

(404) Not found. mostly followed by "Cound not find the controller.". You are calling wrong endpoint or wrong method - eg. using GET request in "monitoring-campaign/get-access" endpoint.

(409) Other transaction in progress. Access to data temporary denied. Something else is in critical section for this part of data. Try again later.

(500) Program exception catched. System exception occured. Exception message is display alongside error message.

(503) Service temporarily unavailable. You are requesting too fast. Your request was refused beacuse you reached your requests limit, wait few seconds and try again.

(503) Service unavailable. Connection was disturbed. It can be system/connection temporary failure, or throttle prevention (see below)

(520) Something went wrong. Cant determine what, but something brokes on the way, if you see this type of error let us know.

Misc. info

Changes in API are listed here and all apply to version 3 of SEMSTORM API.

Tags are obsolete. Tags functionallity is disabled, their endpoint is closed and any usage of them is impossible.

Problem with curl. If you get message "Unsupported request content type..." means you are using invalid content type. Curl examples below use JSON input/output format and we encourage to use this solution, if you want to communicate by JSON as well add -H "Content-Type: application/json" in your curl request (see here).

Advanced filters. Some of requests have optional "advanced filters" parameter. Those filters allows user to create advanced and precise reports/queries. Those filters are sent in request as simple array, but which must be build properly. Setting them in curl could be pretty troublesome, but for PHP library users there is special class created named SemstormFilters, which makes it a lot simplier. For details about usage of SemstormFilters class see examples/Explorer/FiltersExplain.php and examples/Explorer/FiltersUsage.php files. Currently requests using advanced filters are: Campaign get data.

Variables types: Services backend is implemented in PHP, which does not require explicit type definition (see here), that is why in examples below all variables described as "int" can also be passed as string, and also returned types can differ from described below depend on content-type, and client environment.
For example remember that -eventhou every input and output described below using "array" nomenclature- when returning data in JSON you will get objects in place of assoc arrays.
If your software requires explicit variable type definition remember to cast results into proper variables.

Documentation

Every functionallity contains examples in curl and PHP. In PHP case, it is assumed that user have properly installed SEMSTORM PHP SDK library which can be fount here. For curl -as any other than prepared sdk library- remember that SEMSTORM API guarantee compatibility only for JSON formatted data.

Monitoring Campaign

MonitoringCampaign class instance

MonitoringCampaign class is used to perform operations on SEMSTORM Campaigns, which includes creating, retrieving, and modification of campaigns.
PHP example code:
include_once __DIR__ . '/autoload.php';
use SemstormApi\Semstorm;
use SemstormApi\Monitoring\MonitoringCampaign;
Semstorm::init( __ACCESS_TOKEN__ );
$monitoringCampaign = new MonitoringCampaign();

Create campaign

Input   PHP  CURL 
place type reqired default  description
data["domain"] string yes - Campaign domain value.

Output   RESULT 
place type description
result["id"] integer Id of newly created campaign.

Endpoint: /monitoring/monitoring-campaign [POST]

Description:
Creates new campaign.
Can create only "seo" type campaigns.

Error:
Request: domain field is empty.
Limit: creating campaign require at least one free campaign limit.

PHP example code:

examples/Monitoring/CampaignCreate.php

Curl:

curl -X POST -d '{ "domain": "example.com" }' http://api.semstorm.com/api-v3/monitoring/monitoring-campaign.json?services_token=__ACCESS_TOKEN__

Curl output "result":

{ "id": __ID__ }

Get list of campaigns

Input
place type reqired default  description
data["type"] string no seo Type of campaigns to retrieve. Proper values are "seo", "ads", "trash".
Seo and ads contains active and stopped campaigns while trash list deleted ones.
data["pager"]["items_per_page"] integer no 25 Page size (how many campaigns are in one page).
data["pager"]["page"] integer no 0 Page number to retrieve, starting from 0.
data["filters"]["engine"] mixed no All Engine id (int) or "All" (string) for all engines.
data["filters"]["status"] array no Array with both statuses. Array with statuses (strings), possible values: "active", "stop".
data["filters"]["bookmark"] mixed no All 0 - non bookmarked campaigns. 1 - bookmarked campaigns. 'All' - both.
data["filters"]["title"] string no Empty string. Title or part of title to filter.
Output   RESULT 
place type description
result["count"] int Number of all campaigns (for better pagination).
result["campaigns"] array Array with campaigns.
result["campaigns"][]["id"] int Campaign id.
result["campaigns"][]["access"] string User access type.
result["campaigns"][]["status"] string Campaign status, possible values: "active", "stop", "delete".
result["campaigns"][]["domain"] string Campaign domain.
result["campaigns"][]["type"] string Campaign type, possible values: "seo", "ads", "trash".

Endpoint: /monitoring/monitoring-campaign/get-list [POST]

Description:
Get campaigns set selected by pager.

Error:

Input
place type reqired default  description
data["id"] integer yes - Campaign id to retrieve.
data["domains"] mixed no Campaign domain Array of domains or string with one domain.
data["type"] string no organic Type of data to retrieve, valid values are "organic", "paid" and "maps".

Get campaigns access

Input
place type reqired default  description
Output   RESULT 
place type description
result array Array with campaigns access informations, keyed by campaigns ids.
result[_CAMPAIGN_ID_]["access"] string Your access level, possible values are: "owner", "manager", "readonly".
result[_CAMPAIGN_ID_]["owner"] string Owner e-mail address.

Endpoint: /monitoring/monitoring-campaign/get-access [POST]

Description:
Returns information about campaigns you have access to, with type of access and infrmation about owner (e-mail address).
Existing permissions are:
"owner" - this account can modify everything in campaign, add and remove roles. Its reserved for account which created campaign, and is persistent.
"manager" - this account can do everything as "owner" but this role can be removed.
"readonly" - account with that permission can only see campaign and its statistics, and cannot do any changes.

Error:
none.

{ "result": { (see php example for example result) } }

Set campaigns access

Input
place type reqired default  description
data array yes - Array with campaigns access modifications, keyed by campaigns ids.
data[__CAMPAIGN_ID__]["set"] array no Empty array. Array with accesses to set for given campaign (add or modify existing).
data[__CAMPAIGN_ID__]["set"][]["mail"] string yes - User e-mail for whom access will be set.
data[__CAMPAIGN_ID__]["set"][]["permission"] string yes - Permission level to set, possible values are: "readonly", "manager".
data[__CAMPAIGN_ID__]["remove"] array no Empty array. Array with accesses to remove for given campaign.
data[__CAMPAIGN_ID__]["remove"][]["mail"] string yes - User e-mail for whom access will be removed.
Output   RESULT 
place type description
result array Array with access modifications results, keyed by campaigns ids.
result[_CAMPAIGN_ID_]["set"][_USER_MAIL_] bool True if change was successful, false otherwise.
result[_CAMPAIGN_ID_]["remove"][_USER_MAIL_] bool True if removal was successful, false otherwise.

Endpoint: /monitoring/monitoring-campaign/set-access [POST]

Description:
Set access to campaigns, you need to have proper access level to modify permissions in campaigns.
This api endpoint is limited to one request per 10 seconds.
By this api call you can alter maximum 3 accounts at once. That means "set" array shouldn't have more than 3 records. There is no limit in removing access.
Proper permissions to give and take are "readonly" and "manager". One more permission named "owner" is reserved for account which created campaign and cannot be modified.
If you will try to grant permission by entering e-mail on which there is no registered account in SEMSTORM system, then the invitation e-mail to craete account will be send.

Error:
Request: too fast requesting, if any campaign id invalid (user might not have permissions) or if access updating failed. Access updating will fail if: you try to modify permissions in campaign you have "readonly" access, you try to remove or change permission level of the owner of campaign, one or more e-mail address is invalid.
Throttling: this request is specially limited to one per 10 seconds.

PHP example code:

examples/Monitoring/CampaignSetAccess.php

Curl:

curl -X POST -d '{ "__CAMPAIGN_ID__": { "set": [ (...) ], "remove": [ (...) ] }}' http://api.semstorm.com/api-v3/monitoring/monitoring-campaign/set-access.json?services_token=__ACCESS_TOKEN__

Output:

{ "result": { "__CAMPAIGN_ID__": { "set": [ (...) ], "remove": [ (...) ] } } }

Throttle prevention errors

Error with code 503 in most cases will concern throttle prevention. If error message will contain "Throttle prevention. Data may be incomplete(...)" that means system was overloaded and refused to sustain connection.
This is temporary problem, you can retry your request after few secons - just remember that very large request will require more time to wait.
If problem sustains it propably means you are requesting too big set of data at once. Try split it into few smaller requests.
